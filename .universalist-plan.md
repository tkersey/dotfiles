# Universalist Plan

## Track

Track I — Effective Universal Software Synthesis, constrained to two coupled
witness seams: a single authority-bound actuation step and pure validation of
the governance artifacts that feed it.

## Signal

The actuation kernel is causal and Zig-native, but its review path still reached
a Python RF-v2 validator and `$plan` still reached Python PSC-v1/PSR-v1
validators. The remaining signal is duplicated interpreters at the governance
boundary: validation must be pure and Zig-native without becoming authority.

## Construction

Use an append-only, hash-chained actuation event store at `.ledger/actuation/events.jsonl` and expose it through the existing `ledger --source SOURCE` dispatcher as `ledger --source actuation`.

Expose the three stateless governance checks through a separate command family:

```text
ledger validate {plan-source-contract,policy-synthesis-receipt,review-fold}
```

`validate` is a pure observation over canonical JSON. It neither reads nor
writes `.ledger`, and its decision explicitly grants no authority.

`/goal` remains the outer fixed-point driver. One CLI invocation performs at most one kernel transition:

```text
coalgebra: current state -> next admissible operation or terminal condition
handler:   prepared capability -> concrete effect or external-effect observation
algebra:   prior state + immutable event -> next state

/goal:     repeat the one-step kernel until close or block
```

The ledger fold is a catamorphism over immutable events. Next-operation admission is the coalgebraic projection. Their effectful composition is hylomorphic in the workflow, but the repo-local CLI does not own or conceal the recursive controller.

## Canonical boundary artifact

An `actuation-event/v1` chain whose fold yields exactly one `actuation-kernel-state/v1`. A prepared operation carries a random, single-use capability digest bound to the run, step, obligations, exact paths, verifier, and live repository state.

Boundaries:

- GoalContract -> kernel run: obligation compiler and authority binding.
- Kernel command -> runtime: effect handler or explicitly external edit boundary.
- Runtime -> observation: independent repository snapshot and kernel-executed verifier.
- Event chain -> state: total fold with sequence, predecessor-digest, transition, and capability-use checks.
- State -> `/goal`: one-step result and next admissible transition.
- Governance artifact -> validation decision: one total, stateless Zig
  interpreter selected by contract kind.

## Boundary kind

Interpreter plus behavioral coalgebra, with an explicit capability-sensitive pure/effect boundary.

## Composition geometry

Ordered effectful composition. Repository mutation, observation, and closure do not commute. Only read-only state projection is central; capability issue, capability consumption, verification, and closure remain causally ordered.

## Boundary law

- Obligation conservation: closure succeeds only when every accepted obligation has a successful kernel observation.
- Causal admission: an edit is recordable only from a capability issued against the exact pre-effect state.
- Single use: one capability digest and one idempotency identity can authorize at most one effect transition.
- Replay integrity: each event sequence and predecessor digest must extend the current chain exactly once.
- Observation independence: the kernel executes the admitted verifier and computes repository/path state itself.
- Fixed-point ownership: the CLI performs one transition; `/goal` alone decides whether to iterate.
- Validation purity: the same canonical JSON yields the same sorted decision,
  no validator mutates storage, and no pass verdict grants execution authority.

Strongest falsifier: a post-hoc, replayed, out-of-scope, verifier-substituted, or obligation-free edit reaches `closed`.

## Freyd/AFT boundary diagnostic

Not a free-builder problem. The central issue is ordered effects and causal authority, so the relevant diagnostic is the pure/effect boundary: pure state folds may compose freely; effectful transitions may not be reordered, duplicated, or discarded.

## Composition Certificate

- Interpreter: the Zig `actuation` ledger source module.

- Bypass policy: a repo-local CLI cannot physically intercept Codex's `apply_patch` primitive. External edits are therefore admitted before the effect and independently reconciled afterward; unmediated effects remain an explicit obstruction rather than a claimed guarantee.

Current world: mutable YAML receipts whose temporal meaning is inferred after the fact.

Easy world: first-order operation syntax plus an append-only event algebra.

Transfer law: every accepted operation syntax denotes exactly one legal state transition, and every terminal claim is reconstructed from the chain rather than trusted from the caller.

Falsifier: two distinct event histories with different authority, effects, or observations fold to the same closed state.

## Why this construction

- Extending `ledger --source SOURCE` uses a live, tested namespace boundary that already exists.
- Adding `ledger validate` keeps stateless observations out of `--source`, whose
  namespaces own state and event folds.
- A new standalone controller would duplicate `/goal` and violate fixed-point ownership.
- Adding more deterministic receipt fields would improve description without establishing causal order or single use.
- Retiring the Python gates removes duplicate interpreters. Reusable actuation
  behavior is expressed as verifier-backed obligations plus the Zig `decide`
  projection; PSC-v1, PSR-v1, and RF-v2 retain pure invariant checks under
  `ledger validate`. Dominated policy selectors, checkpoints, and transition
  receivers are retired rather than transliterated into a second controller.

## Effective Universal Architecture Certificate

- Program representation: typed kernel commands and `actuation-event/v1` JSONL rows.
- Evaluator/interpreter: Zig command validator, effect handler, event appender, and total fold.
- Recursion/partiality: `/goal` supplies unbounded iteration; the CLI exposes one partial transition with explicit blocked errors.
- State/interaction: event-sourced run state with one pending operation at most.
- Concrete primitives: cryptographic randomness, Git/filesystem observation, process execution, wall-clock annotation, durable lock, and append-only storage.
- Observational equivalence: histories are equivalent only when authority, obligation status, artifact fingerprint, pending capability state, and terminal outcome agree.
- Resource model: serialized append under a lock; bounded input/store/process output; repository fingerprint work linear in current visible diff plus untracked content and admitted paths.
- Capability model: 256-bit capability material is returned once and only its digest is persisted; token transport confidentiality remains caller/runtime-owned.
- Obstruction: complete mediation of in-app Codex mutation tools is unavailable to a repo-local process. The witness establishes causal admission plus independent reconciliation, not an OS sandbox.
