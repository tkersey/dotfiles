# Universal Architecture Kernel

Universal architecture is the practice of designing software around canonical boundary artifacts and, when two-dimensional change, locality, indexing, or context action is semantic, explicit structure inside worlds and their process/change, description, and generalized-morphism categories.

Central rule:

```text
Primitive operations may be arbitrary.
Composition boundaries may not be arbitrary.
Process composition and architecture-change composition may not be conflated.
Local context may not be implicit when it changes correctness.
The composition of descriptions may not be invented independently of their indices.
Context framing may not be duplicated independently of its action laws.
```

Ordinary code may live inside a boundary: I/O, arithmetic, parsing, database drivers, vendor APIs, ML inference, clocks, randomness, local algorithms. The boundary where such code composes with other worlds should be explicit and testable.

## Kernel artifacts

1. **Worlds**
   - Domain, API, DB, events, UI, runtime, compiler IR, policy, tests, observability.

2. **Boundaries**
   - Embeddings, projections, handlers, interpreters, serializers, compilers, views, report extractors, contextual frames, optics, migrations, and compatibility squares.

3. **Free syntax**
   - ASTs, commands, workflows, effect programs, query languages.

4. **Coherent observations**
   - Sanctioned projections, old client views, report observations, policy queries.

5. **Transported semantics**
   - Old behavior moved to new target surfaces through explicit paths.

6. **Lifted implementations**
   - Public behavior determines internal realizer or implementation template.

7. **Free builders behind projections**
   - A disciplined observation boundary supports canonical construction back into implementation space.

8. **Obstruction reports**
   - The projection loses evidence, constraints cannot combine, templates are unbounded, locality has no effective presentation, description decompositions/quotients are non-effective, context framing lacks an effective action/realizer, or a square calculus cannot paste/interchange effectively.

9. **Behavioral coalgebras**
   - Ongoing behavior specified by transition plus observation.

10. **Effect signatures and handlers**
    - Operation syntax separated from runtime interpretation.

11. **Base composition geometry**
    - Categories, monoidal/Freyd structures, operads, PROPs, traced structures, and resource-sensitive worlds make legal composition explicit.

12. **Two-dimensional composition**
    - Double categories separate horizontal process/generalized-arrow composition from vertical migration/refinement/strict-map composition; typed squares, both pastings, and interchange make compatible local change compositional.

13. **Equipments and base change**
    - Tight maps induce companions, conjoints, or restrictions of loose generalized arrows only when their unit/counit, base-change, effectivity, and failure laws are explicit.

14. **Description composition**
    - Pointwise, Day, promonoidal, substitutional, or monadic products are selected separately for functors, presheaves, predicates, plans, graded families, rules, contexts, or spatial descriptions.

15. **Day/promonoidal convolution**
    - Base composition is lifted to indexed descriptions through legal decomposition witnesses, coherent quotienting, interpretation, and effective enumeration/normalization.

16. **Context actions and Tambara modules**
    - An ambient context world acts on endpoint worlds; a profunctor receives a lawful frame operation with unit, associativity, naturality, coherence, interpretation, and resource laws.

17. **Optic/residual-context IRs**
    - A focus is decomposed with residual context and rebuilt through a Tambara-aware interpreter; residual reindexing is quotiented explicitly.

18. **Free/cofree contextual closure**
    - Bare capabilities may be closed under every legal frame, or observed under every supported frame, using an effective basis or obstruction.

19. **Representability witnesses**
    - A generalized relation becomes a concrete context-preserving functor/adapter only when a realizer or right-adjoint witness exists.

20. **Comonadic spatiality**
    - Points have coherent local neighborhoods; density comonads generate locality; coalgebras represent situated objects; halos and germs expose local context; continuous boundaries preserve locality.

21. **Effective spatial presentations**
    - Finite, bounded, indexed, or queryable halos; explicit local/global identity; labelled relationships; basis/reconstruction or obstruction; invalidation and resource laws.

22. **Spatial convolution and framing**
    - External-product patches compose suitable spatial descriptions; Tambara actions may frame transformations through lawful halo/context extension.

23. **Explicit IR**
    - Callbacks, handlers, continuations, predicates, mappers, rules, residuals, context frames, migrations, and compatibility squares become data plus interpreter.

24. **Law tests**
    - Executable witnesses for construction, observation, projection, lowering, transition, handler, interpreter, two arrow compositions, square boundaries, pasting, interchange, base change, decomposition, quotient, context framing, optic interpretation, center/coherence, restriction, density, or continuity.

## Operating test

A proposed artifact, square calculus, spatial structure, description product, or context action is legitimate only if it changes at least one of:

- code shape or module boundary;
- artifact provenance;
- horizontal/vertical arrow ownership;
- square construction, pasting, normalization, or invalidation;
- index/grade/resource representation;
- legal decomposition or admissibility;
- ambient context/action or residual representation;
- local/global identity and dependency/ownership/capability/evidence context;
- central interpreter/projection/handler/context transport/frame/double-functor operation;
- static analysis, cost, documentation, tests, or multi-interpreter support;
- ability to reject invalid states, illegal decompositions, invalid frames, or mismatched squares;
- ability to compose local migration/refactor witnesses into a global proof;
- ability to distinguish a relation/specification from a real implementation map;
- impact/invalidation/resource accounting.

A category, 2-category, typed adapter plus one witness, PROP, DPO rewrite, product, pointwise map, operadic substitution, monadic program, pullback, pushout, ordinary profunctor, Reader parameter, residual record, dependency graph, or context object remains preferable when it already expresses the intended architecture without double-category, Day, Tambara, or spatial mechanics.
