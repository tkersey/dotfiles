# Comonads as Spaces

Use this mechanics reference after Universalist has identified a boundary or world whose correctness depends on locality, neighborhoods, situated context, restriction, or local-to-global reconstruction.

Research basis: Aaron David Fairbanks, Kevin Carlson, and David I. Spivak, *Comonads as Spaces*, arXiv:2607.15091v1 (2026). The mathematical claims below summarize that paper and standard comonad theory. The software mappings are Universalist architecture inferences.

## 1. Core mathematical structure

A comonad on a category `S` is an endofunctor:

```text
C : S -> S
```

with natural transformations:

```text
delta   : C => C . C    comultiplication
epsilon : C => Id       counit
```

satisfying coassociativity and counit laws.

For a software-oriented comonad:

```text
C(A)       A-values equipped with a local/situated view
epsilon    extract the value at the center
delta      expose a local view at every point of a local view
```

The laws require the center and nested neighborhoods to cohere.

## 2. Coalgebras as situated objects

A `C`-coalgebra is an object `E` with:

```text
h : E -> C(E)
```

such that:

```text
C(h) . h = delta_E . h
epsilon_E . h = id_E
```

Software reading:

```text
Each element is assigned a coherent local neighborhood of elements.
The center of its neighborhood is itself.
Neighborhoods viewed from nearby points agree with nested neighborhood structure.
```

Special cases from the paper:

- for a topological space, coalgebras of the associated germs comonad are sheaves/étalé spaces;
- for a small category, coalgebras of the associated polynomial comonad are covariant functors/actions;
- more general comonads yield generalized spaces and situated structures not reducible to topology or ordinary categories.

Do not conflate this with a behavioral coalgebra `X -> F(X)`, which models transition/unfolding. A comonad coalgebra models coherent contextual placement.

## 3. Density comonads generate spaces

Given:

```text
P : B -> S
```

its density comonad, when it exists, is:

```text
<P> = Lan_P P
```

For small `B` and cocomplete `S`, this is pointwise. On `Set`:

```text
<P>(A) = colim_{P(U) -> A} P(U)
```

The paper interprets `P` as a generalized subbasis and `<P>` as the space generated by it.

Software reading:

```text
B       vocabulary of basic local contexts
P(U)    concrete inhabitants/elements of one patch
<P>     locality structure freely generated by the patch vocabulary
```

Typical patch vocabularies:

- syntax scopes and binding regions;
- module/dependency neighborhoods;
- ownership or capability regions;
- schema neighborhoods;
- evidence/support neighborhoods;
- UI/provider contexts;
- process/network neighborhoods;
- test and invariant regions around code seams.

## 4. Subbasis versus basis

A subbasis generates a density comonad. A **basis** is stronger: the canonical lift:

```text
B -> S^<P>
```

into the coalgebra category must be dense.

That means every coalgebra is a canonical colimit of basic coalgebras. Architecturally:

```text
coverage != basis
examples != basis
component catalog != basis
fixture suite != basis
```

A basis supports canonical reconstruction of every situated object from basic patches.

Engineering approximation of density:

- one public reconstruction/fold API;
- every supported situated object decomposes into basic patches;
- reconstruction after decomposition preserves observations;
- equivalent patch diagrams normalize to one representation;
- missing patch families produce an obstruction report.

## 5. Points, local points, and global points

For a comonad on `Set`, its set of points is:

```text
|C| = C(1)
```

For a density comonad `<P>`, the points are the colimit `|P|`.

Important warning from the paper: an element of `P(U)` is a **local point**. The colimit map may identify several local points as one **global point**. The maps `P(U) -> |P|` need not be injective.

Software consequences:

- preserve scoped symbol identity separately from canonical symbol identity;
- preserve source-local entity references separately from reconciled IDs;
- retain provenance when multiple dependency/evidence witnesses share one target;
- do not use global IDs as a substitute for local context.

Certificate fields:

```text
local point representation
global point representation
local-to-global map
identifications introduced
provenance retained
falsifier for premature collapse
```

## 6. Germs

The paper uses germ language for elements of `C(A)`. For a density comonad, a germ may be represented by:

```text
[f]_x
```

where `x` is a local point in some patch `P(U)` and `f : P(U) -> A`, modulo restriction/reindexing.

Software reading:

```text
A germ is an A-valued fact, behavior, or interpretation
valid in some sufficiently local context around a point.
```

Examples:

- a name resolution valid in one scope;
- an authorization fact valid in one tenant/capability neighborhood;
- a test conclusion valid around one code seam and dependency snapshot;
- evidence supporting a claim inside one task context;
- configuration valid for one deployment neighborhood;
- a UI value valid under inherited providers.

Restriction law:

```text
restrict(germ, smaller-valid-neighborhood)
preserves value/meaning at the center
```

## 7. Halos

A **halo** is the paper's abstraction of the infinitesimal neighborhood of a point. It is generally a formal connected limit of neighborhoods rather than an ordinary finite set.

For a comonad `C` on `Set`, the endofunctor decomposes by points:

```text
C(A) ~= Sum_{x in |C|} A^(Halo(x))
```

where `Halo(x)` is a formal connected limit. Elements of `C(A)` are maps from a halo into `A`—germs.

For a category regarded as a comonadic space:

```text
Halo(c) = arrows out of c
```

An arrow is a witness that another point is near, so two points may be near in multiple ways.

Software halo examples:

```text
symbol:
  callers, callees, scopes, tests, generated artifacts, owners, invariants

endpoint:
  request schema, principal, policy, handlers, persistence, traces, consumers

schema entity:
  foreign keys, mappings, constraints, reports, migrations, provenance

decision/task:
  required claims, evidence, policies, stakeholders, capabilities, freshness

runtime node:
  communication channels, dependencies, failure domains, authority, replicas
```

## 8. Labelled halos

Unlabelled halos may erase where a neighborhood witness goes. Labelled halos retain target-indexed structure.

For software, useful labels include:

```text
target point
edge/relation kind
owner
capability/authority
effect
provenance
trust level
temporal validity
version
resource cost
```

Prefer labelled halos when two relationships with the same endpoints have different architectural meaning.

Labelled-halo law:

```text
A boundary preserves or explicitly translates every required label.
Forgotten labels are named and justified.
```

## 9. Continuous maps versus comonad maps

A central warning of the paper:

```text
ordinary comonad map != continuous map
```

For topological examples, continuous maps induce inverse-image functors on coalgebras/sheaves. The paper defines generalized continuous maps via functors between coalgebra categories with cartesian compatibility over point maps.

Direction matters:

- a continuous map `C -> D` induces maps from source halos toward target halos;
- a comonad map `C -> D` induces halo maps in the opposite spatial direction.

Architecture rule:

```text
Do not say a boundary is continuous merely because a natural transformation of comonads exists.
Name the point map, coalgebra/context transport, cartesian/locality compatibility,
and halo direction.
```

Practical continuity law:

```text
for every point x,
mapHalo(Halo_source(x))
fits within Halo_target(mapPoint(x))
and preserves required labelled observations
```

Falsifiers:

- points map but dependency/authority context is lost;
- a local valid section becomes invalid after transport;
- target halo contains no image for a required source witness;
- labels are collapsed without policy;
- inverse-image/restriction behavior disagrees with the point map.

## 10. Underlying topological and categorical shadows

The paper proves that every comonad on `Set` has:

- an underlying topological space, by reflection with respect to continuous maps;
- an underlying small category, by coreflection.

These are shadows. They may forget structure:

- topology may collapse multiple witnesses of nearness to truth values;
- an underlying category may select arrow-like neighborhood structure but not all generalized spatial information.

Software use:

```text
Topological shadow:
  coarse reachability/openness/impact regions.

Categorical shadow:
  typed paths and witnesses of dependency/nearness.

Full comonadic space:
  local context plus coherence beyond either shadow.
```

Do not replace the full spatial model with a shadow unless required observations factor through it.

### Incremental spatial index

Maintain dependency/provenance edges under change. Record:

```text

```

### Patch basis and reconstruction

```text
decompose : SituatedObject -> PatchDiagram
reconstruct : PatchDiagram -> Result[SituatedObject]
```

Law:

```text
observe(reconstruct(decompose(x))) == observe(x)
```

### Context germ

```text
compileLocalContext : TaskPoint -> ContextGerm
restrictContext     : ContextGerm -> SmallerHalo -> ContextGerm
```

### Obstruction

Return a named obstruction when:

- the halo is unbounded under required observations;
- patch coverage is not dense;
- local/global identity is ambiguous;
- continuity cannot be maintained;
- invalidation cost violates the resource model;
- labels required for correctness are unavailable.

## 15. Law and falsifier checklist

```text
Comonad data:
  endofunctor/context former
  counit/extract
  comultiplication/duplicate

Center law:
  extracting the center returns the original value

Neighborhood coherence:
  nested local views agree associatively

Coalgebra law:
  assigned local context is centered and coherent

Restriction/germ law:
  smaller valid neighborhoods preserve local meaning

Basis-density law:
  situated objects reconstruct canonically from basic patches

Continuity law:
  boundary preserves required halos and labels

Local/global identity law:
  identifications are explicit and provenance survives

Resource law:
  halo computation/invalidation stays within declared budget
```

Every claim needs a negative witness such as:

- local context changes after duplicate/restrict;
- point preserved but owner/capability/provenance neighborhood lost;
- two local points silently collapsed;
- examples cover cases but cannot reconstruct a valid situated object;
- finite halo approximation omits a dependency that changes an observation;
- a supposed continuous boundary maps a valid germ to an invalid one.

## 16. Selection and rejection

Use comonadic spatial mechanics when:

- locality is part of correctness;
- points and patches are identifiable;
- nested local context must cohere;
- a basis/reconstruction question exists;
- a boundary must preserve neighborhoods, not only values;
- an effective halo representation changes code or tests.

Reject or narrow when:

- a plain dependency graph is sufficient;
- no counit/comultiplication or locality law can be named;
- “halo” is only a poetic synonym for nearby files;
- the proposed basis is only an example list;
- the infinite spatial model has no effective approximation;
- the architecture recommendation depends on treating an early preprint as settled software practice.
