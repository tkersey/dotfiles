# Universal Problem IR

`universal-problem/v6` is Universalist's evidence contract for using the categorical corpus as a hidden architecture optimizer.

It is deliberately **not** a proof assistant, source analyzer, route authority, implementation authorization, or Skill Decision Receipt. The root workflow inspects repository reality, resolves evidence, adjudicates alternatives, selects the route, and owns any consequential Ledger/Seq receipt. The compiler validates one packet, evaluates compatible theorem cards, exposes proof obligations and evidence debt, and may nominate at most one construction.

## Governing rule

```text
one owned seam
  -> one architectural axis
  -> one typed hole
  -> one Universal Problem packet
```

Independent pressures are not collapsed into one global winner. Data shape, base composition, indexed-description composition, context action, locality, transport/realization, presentation, behavior, schema/context, syntax/semantics, and proof are separate axes. When one seam needs several axes, compile separate linked packets under the same root plan and decision boundary.

## Pipeline

```text
repository evidence
  -> boundary disposition
  -> ordinary repository-native candidate
  -> evidence-bound comparison universe
  -> one axis, one typed hole, and exact requirements
  -> attributed structural signals and typed prerequisite facts
  -> candidate requirement witnesses
  -> candidate proof packets and evidence-bound alternative rejections
  -> theorem-card evaluation
  -> nomination, verification, underdetermination, conflict, unclassified hole, or evidence debt
  -> root adjudication
  -> repository-native lowering or obstruction
```

Category theory remains in the theorem registry and linked doctrine. Default certificates use plain engineering terms. `--explain-theory` adds expert names, universal-property guidance, and source references without changing eligibility or nomination.

## Authority boundary

```text
compiler owns
  packet validation
  one-axis and typed-hole admission
  known-signal and known-prerequisite validation
  theorem-card compatibility
  prerequisite epistemic state
  exact requirement-level witness coverage
  evidence-bound proof-profile completeness
  evidence-bound alternative rejection
  deterministic nomination or verification status
  material-delta projection from witnesses
  evidence-debt reporting

root workflow owns
  repository inspection
  whether evidence locators resolve and support their claims
  comparison-universe policy
  ordinary candidate
  architecture adjudication across linked packets
  final route and implementation authorization
  Ledger/Seq decision receipt
```

A compiler nomination is decision evidence. It is not the decision.

## Attributed claims

Architectural claims use:

```json
{
  "claim": "authorization owns checked tenant agreement",
  "evidence": [
    "code:src/auth/tenant.ts#make",
    "test:tenant-mismatch"
  ]
}
```

Admitted evidence schemes are declared by the registry. Version 6 admits:

```text
code test doc trace schema command artifact requirement
```

The compiler validates syntax and scheme only. The root must verify that locators resolve and support the claim. `assumption:`, `not-inspected:`, prose, and bare filenames cannot discharge evidence.

## Packet header

Every packet declares:

```json
{
  "schema": "universal-problem/v6",
  "claim_mode": "nomination",
  "claim_scope": "effective_realization",
  "boundary_disposition": "introduced",
  "disposition_rationale": {
    "claim": "the endpoint introduces a new owned boundary",
    "evidence": ["requirement:boundary-owner"]
  },
  "verification_target": null
}
```

Claim modes:

```text
nomination    compare possible architecture before root selection;
              verification_target must be null
verification  verify one explicitly named selectable theorem card;
              verification_target must be that card id
```

Verification checks evidence for a named construction. It never nominates a route or emits a root decision evidence readiness. Competing attributed alternatives and active obstructions still have to be resolved.

Claim scopes:

```text
literal                the repository explicitly realizes the stated mathematical structure
                       and its required comparison maps/laws
effective_realization  repository-native code realizes the admitted universal obligations within
                       a stated observational and resource model
bounded_approximation  the implementation intentionally realizes only a declared bounded fragment;
                       approximation_boundary and refinement_condition proofs are mandatory
```

Analogy alone is not a claim scope and cannot nominate a construction. It belongs only in explanatory prose.

Boundary dispositions:

```text
preserved
introduced
changed
repaired
removed
bypass-justified
```

Disposition controls ordinary routing:

```text
preserved                         -> UNI-PRESERVE
introduced / changed / repaired  -> UNI-ORDINARY when the ordinary candidate is sufficient
removed                           -> UNI-ORDINARY when the removal law is sufficient
bypass-justified                  -> UNI-OBSTRUCT policy route
```

The compiler does not infer `preserved` merely because no advanced construction is retained. Every disposition has an attributed rationale. `bypass-justified` therefore requires evidence for the bounded primitive exception rather than relying on the enum value alone.

## Comparison universe

Every packet states evidence-backed:

```text
objects
transformations
observations
equivalence
effects
authority
resources
```

These are architecture policy. They determine which alternatives are admissible and what preservation, mediation, uniqueness, and effectivity mean. A construction cannot be universal in the abstract; it is universal relative to this declared universe.

## Architectural axis and hole

The axis is one of:

```text
data-shape
syntax-semantics
behavior

```

The hole kind is one of:

```text

interpreter
composition
representation
equivalence
locality
context-action
proof
```

A packet admits only signals whose owning theorem card supports the declared axis, hole kind, claim mode, and claim scope. An unknown signal fails closed. A signal belonging to another axis fails with an instruction to create a separate linked packet.

The hole carries exact requirements:

```json
{
  "id": "AUTH-OWNER",
  "dimension": "owner",
  "claim": "authorization owns compatible construction",
  "evidence": ["requirement:auth-owner"]
}
```

Material dimensions are:

```text
owner

compatibility
representable_states
legal_composition
effect_semantics
locality
information_flow
interpreter
construction_paths
proof
resources
existence

```

## Facts and epistemic state

Prerequisite facts use exact states, not language truthiness.

### Evidenced true

```json
{
  "status": "evidenced",
  "satisfied": true,
  "value": "tenant id",
  "evidence": ["code:src/auth/tenant.ts"]
}
```

### Evidenced absent

```json
{
  "status": "absent",
  "satisfied": false,
  "evidence": ["test:no-index-tensor"]
}
```

### Unknown

```json
{
  "status": "unknown",
  "satisfied": null,
  "reason": "the decomposition owner was not inspected",
  "evidence": []
}
```

The combinations are exact. Strings such as `"yes"` and `"false"` cannot satisfy a fact. Unknown prerequisite keys are rejected. Facts unrelated to this packet's attributed signals are rejected. Missing or unknown evidence creates evidence debt, not an architectural obstruction.

## Ordinary candidate

The ordinary candidate is stated first:

```json
{
  "artifact": "checked pair helper",
  "law": {
    "claim": "tenant identifiers match",
    "evidence": ["test:tenant-match"]
  },
  "falsifier": {
    "claim": "a mismatch is accepted",
    "evidence": ["test:tenant-mismatch"]
  },
  "resource_impact": {
    "claim": "constant-time comparison and no new invalidation surface",
    "evidence": ["command:benchmark-auth"]
  },
  "guarantees": []
}
```

The caller does **not** submit `material_delta`. The compiler derives delta only from exact candidate witnesses for requirements the ordinary candidate does not already guarantee.

## Candidate requirement witnesses

A candidate witness is specific to one card and one unmet requirement:

```json
{
  "requirement_id": "AUTH-OWNER",
  "dimension": "owner",
  "claim_mode": "nomination",
  "claim_scope": "effective_realization",
  "claim": "authorization becomes the only public constructor",
  "evidence": ["code:src/auth/tenant.ts#make"]
}
```

A witness is rejected when it:

- targets an unknown requirement;
- targets a requirement already guaranteed by the ordinary candidate;
- uses a different material dimension;
- claims a dimension the theorem card does not guarantee;
- uses a different claim mode or claim scope;
- targets a card incompatible with the packet axis or hole;
- targets a support-only card;
- lacks a matching attributed signal.

## Candidate proof packets

Requirement witnesses establish material delta. They do not prove universality. Each selectable card also declares a proof profile.

### Universal profile

```text
existence
commutation
mediation
canonicality
effectivity
falsifier
```

### Obstruction profile

```text
nonexistence
counterexample
stability
effectivity
falsifier
reopening_condition
```

Each obligation is an attributed claim:

```json
{
  "mediation": {
    "claim": "every compatible producer converts through the authorization owner",
    "evidence": ["test:compatible-producer-factorization"]
  }
}
```

Missing proof obligations keep the candidate incomplete. Generic theorem-card prose is guidance, not repository proof.

For `bounded_approximation`, every universal or obstruction profile is extended with:

```text
approximation_boundary  what the bounded realization includes, excludes, and may lose
refinement_condition    what evidence or resource change permits a stronger realization
```

## Candidate rejections

When several attributed candidates remain, the compiler may nominate one only after every other matched candidate is contradicted by prerequisite evidence or explicitly rejected:

```json
{
  "validated_subtype": {
    "basis": "requirement-mismatch",
    "claim": "the domain is a genuinely exclusive constructor family, not one stable predicate",
    "evidence": ["test:exclusive-constructor-counterexample"]
  }
}
```

A rejected candidate cannot simultaneously carry requirement witnesses or proof claims. Rejection bases are typed: `counterexample`, `prerequisite-absent`, `observation-failure`, `requirement-mismatch`, `resource-dominance`, `ordinary-dominance`, or `scope-mismatch`. Signal count, evidence count, registry order, and diagnostic priority never manufacture a winner.

## Theorem cards

Every registry card declares:

```text
selection mode: selectable or support_only
compatible axes and hole kinds
claim modes and claim scopes
proof profile
material dimensions it can guarantee
unique structural signals
prerequisite facts
plain repository artifact
laws and falsifiers as theorem guidance
competitors and mediator
canonicality / uniqueness-up-to
effectivity expectation
boring fallback
references into the retained categorical corpus
```

Support-only cards define or guard the comparison universe. They always use route `NONE`, guarantee no material delta, use the support proof profile, and can never become the implementation artifact.

### Derived mechanics and linked packets

Some expert concepts are not additional product artifacts. They are compositions or guards across independent axes and are recorded in `derived_mechanics` rather than entered into one global theorem-card race:

- category pivot = linked presentation and transport/realization packets;
- spatial description product = locality packet plus description-composition packet;
- spatial context framing = locality packet plus context-action packet;
- strong context-action / Day-center claims = linked description, context-action, and proof packets under explicit closedness or rigidity evidence;
- Cayley-style representation = context-action plus representability/effectivity proof;
- primitive presentation = an ordinary handler/observation boundary, not a manufactured categorical artifact.

Each linked packet is independently nominated or verified. Their root synthesis may select a combined architecture only after every axis is resolved; the compiler never collapses the combination into one winner.

## Nomination and verification law

```text
ordinary candidate covers every requirement
  -> ordinary-sufficient
  -> route from boundary disposition, not automatically preserve

unmet requirements with no compatible selectable signal
  -> unclassified-hole
  -> no route and no selected artifact

all attributed alternatives contradicted/rejected while requirements remain unmet
  -> no-admissible-construction
  -> no ordinary fallback pretending the hole is solved

active obstruction plus active constructive candidate
  -> evidence-conflict or obstruction-unresolved

one complete obstruction and no constructive alternative remains active
  -> obstruction nomination

more than one complete constructive candidate
  -> underdetermined

one complete constructive candidate plus any unresolved alternative
  -> alternative-unresolved

one complete constructive candidate and all alternatives contradicted/rejected
  -> nominated

relevant card with missing/unknown prerequisites
  -> epistemically-blocked

relevant card with incomplete requirement witnesses or proof profile
  -> evidence-incomplete

verification target complete and every alternative resolved
  -> verified
  -> verified_route reported, route_nomination remains null

verification target incomplete or contradicted
  -> verification-failed

verification target conflicts with obstruction or unresolved alternative
  -> verification-conflict / verification-alternative-unresolved
```

Support-only signals may enrich the diagnostic record, but they never solve an unmet hole or create a route.

## Obstruction law

`UNI-OBSTRUCT` requires:

- an attributed signal for witnessed nonexistence, inconsistency, nonrepresentability, resource lower-bound failure, or canonical underdetermination;
- an evidenced obstruction counterexample;
- an evidenced reopening condition;
- an exact witness for an `existence` or `proof` requirement;
- a complete obstruction proof profile;
- no unresolved constructive alternative.

Missing, unknown, uninspected, or merely failed implementation evidence is never an obstruction. A complete obstruction and a complete constructive candidate create an evidence conflict that the root must adjudicate.
