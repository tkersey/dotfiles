# Comonadic Spatiality Doctrine

## Status and claim boundary

This doctrine is a Universalist engineering interpretation of Fairbanks, Carlson, and Spivak, *Comonads as Spaces* (arXiv:2607.15091v1, 2026). The paper develops mathematics; the software-architecture rules below are proposed applications. Mark them as architecture inference unless the codebase explicitly models the comonad, coalgebras, basis, halos, and continuity laws.

## Thesis

A Universalist world is normally described by objects, transformations, invariants, observations, primitives, and composition. When **locality** is semantic, enrich the world with a comonadic spatial structure:

```text
points                 things that may be situated
local patches          neighborhoods, scopes, dependency regions, contexts
subbasis P : B -> Set  chosen local generators
space <P> = Lan_P P    density comonad generated by those patches
coalgebras             coherently situated objects
halos                   infinitesimal/local neighborhoods of points
continuous boundaries  maps preserving required locality
```

Maxim:

```text
Worlds have halos.
Context is a germ.
Boundaries preserve locality.
```

This is optional structure. Do not spatialize a world merely because it has dependencies.

## What problem it solves

Use Comonadic Spatiality when architecture is wrong because it models only what components **globally do** and not what they may **locally see, depend on, inherit, affect, or justify**.

Signals include:

- a change appears local but its dependency, ownership, policy, test, or provenance neighborhood is implicit;
- boundaries preserve entities or values but destroy their surrounding context;
- context is assembled globally and trimmed instead of constructed around a task point;
- local and global identity are collapsed too early;
- a fixture/component catalog covers examples but cannot canonically reconstruct situated objects;
- a refactor preserves outputs while changing the neighborhood that makes those outputs valid;
- dependency edges have several meanings but are collapsed to unlabelled adjacency;
- impact analysis has no law beyond transitive graph reachability.

## Spatial dictionary for software

| Spatial term | Software reading |
| --- | --- |
| Point | symbol, module, endpoint, state, schema entity, task, decision, capability, actor, component |
| Local patch | scope, dependency slice, ownership region, evidence bundle, capability region, test neighborhood |
| Local point | a point as represented inside one patch, retaining local provenance |
| Global point | canonical/coarse identity after colimit or reconciliation |
| Germ | value, behavior, or evidence valid in some sufficiently small neighborhood |
| Halo | formal/effective neighborhood of a point, including multiple witnesses of nearness |
| Labelled halo | neighborhood whose edges retain target, kind, owner, effect, provenance, trust, or time labels |
| Coalgebra | object equipped with coherent situated/local structure |
| Subbasis | local generators sufficient to induce a candidate spatial structure |
| Basis | generators dense enough for canonical reconstruction of all situated objects |
| Continuous boundary | boundary whose point map also preserves required local structure |

## Density before gluing

Density comonads and sheafification solve different problems:

```text
Density comonad:
  generates the geometry of locality from chosen patches.

Possibility Sheafification:
  glues compatible local meanings inside that geometry.
```

Do not call a set of examples a basis merely because it covers test cases. A basis claim requires a canonical reconstruction/density witness or an explicitly bounded approximation.

## External-product locality and conditional Day comparison

Density, indexed description composition, and sheafification are separate obligations:

```text
Density generates locality.
A specified Day or promonoidal product composes indexed descriptions of locality.
Sheafification glues compatible meaning within locality.
```

For two patch vocabularies:

```text
P1 : B1 -> Set
P2 : B2 -> Set
```

the always-defined engineering starting point is the external-product patch vocabulary:

```text
P1 external-product P2 : B1 x B2 -> Set
(U,V) |-> P1(U) x P2(V)
```

and its density construction `<P1 external-product P2>`.

Do not write `<P1> star <P2>` until both spatial descriptions are placed in one specified monoidal or promonoidal description category. A literal or effective Day comparison requires:

```text
shared/product index world B
monoidal tensor and unit, or explicit promonoidal kernel, on B
reindexing of both spatial descriptions into [B,V]
Day_B(<P1>, <P2>)
density-Day comparison map Day_B(<P1>, <P2>) -> <P1 external-product P2>
proof that the comparison is an isomorphism, observational equivalence,
or explicitly bounded approximation
```

Without those data, call the result an external-product density construction rather than a Day product.

Software interpretation:

```text
left patch vocabulary
x
right patch vocabulary
  -> external-product patches
  -> combined points/halos/labels
  -> continuous projections back to both worlds
  -> optional Day comparison only with a shared index witness
```

Potential uses:

- dependency locality combined with capability locality;
- evidence locality combined with freshness/trust locality;
- ownership locality combined with deployment locality;
- syntax-scope locality combined with type/effect locality;
- task-context locality combined with policy locality.

Do not implement this as simple list concatenation unless concatenation satisfies the required reconstruction, halo, label, continuity, and resource laws. For arbitrary comonadic spaces the relevant Day tensor need not be the ordinary cartesian product of point sets.

A spatial-composition certificate records:

```text
left and right subbases/bases
shared/product index world
tensor/unit or promonoidal kernel
external-product patch vocabulary
reindexing into one description category
density-Day comparison map
isomorphism / observational-equivalence / bounded-approximation / no-Day-claim status
combined point representation
combined halo representation
label product/translation
continuous projections
local/global identity policy
finite/indexed/incremental implementation
collision and resource falsifiers
```

## Context as a germ

Exact Context may use this spatial refinement:

```text
semantic consumer / task = point q
relevant dependencies    = Halo(q)
prepared Context(q)       = a certified section or germ over an effective halo
rendering                 = a consumer-specific presentation of that germ
```

This changes the question from:

```text
Which globally retrieved facts should we trim?
```

to:

```text
Which local patch around q supports the required observations,
and how is validity preserved under restriction and transport?
```

A spatial context certificate records the task point, halo approximation, local/global identities, restriction rules, labels, and continuity law.

When task requirements themselves form a monoidal/promonoidal index world, context fragments may additionally compose by Day/promonoidal convolution. That is optional description-level structure and does not replace context schema, provenance, freshness, constraint closure, or germ validity.

## Continuous boundaries

Preserving points is weaker than preserving locality. A boundary may map every entity successfully while losing its callers, permissions, provenance, policy neighborhood, or evidence conditions.

A locality-sensitive boundary must specify:

```text
point map
source and target spatial worlds
map or approximation between halos
labels preserved, translated, forgotten, or generated
restriction/germ behavior
continuity law
continuity falsifier
```

Do not identify an ordinary comonad morphism with a continuous map. In the paper these are distinct structures and induce halo maps in opposite directions. For architecture, name which notion is intended.

## Effective presentation

The Substrate Reality Law still applies. A formal halo may be an infinitary formal limit; production software needs one of:

- a finite exact neighborhood;
- a bounded-depth or budgeted approximation;
- an incremental dependency index;
- a query/view that computes the halo on demand;
- a labelled graph or hypergraph representation;
- a compact basis whose reconstruction is executable;
- an obstruction report stating why no effective halo/basis is available.

Record complexity, invalidation, freshness, and cache consistency. A beautiful locality model that cannot be maintained is not an architecture.

For spatial convolution, additionally record the size of the external-product patch system, pair enumeration strategy, normalization/collision policy, and whether the combined halo can be maintained incrementally.

## Law catalogue

### Center / counit law

```text
extract(localView(x)) == valueAt(x)
```

The center of the neighborhood is the original point/value.

### Neighborhood coherence / comultiplication law

```text
localViewOfLocalViews(x)
  agrees with
expandLocalView(localView(x))
```

Nested neighborhoods compose coherently; local context does not change merely because it is observed from a nearby point.

### Restriction / germ law

```text
restrict(section, smallerValidHalo)
preserves local meaning at the center
```

### Continuity law

```text
mapHalo(Halo_source(x))
  is accepted inside
Halo_target(mapPoint(x))
```

Required labels and observations survive the boundary.

### Basis-density law

```text
every situated object is canonically reconstructed
from its diagram of basic patches
```

In engineering, approximate uniqueness with one reconstruction API, normalization, and no unchecked competing construction path.

### Labelled-halo law

```text
edge kind, destination, owner, capability, effect,
provenance, trust, and temporal labels survive or are translated explicitly
```

### Conditional spatial Day representable law

Only after a shared/product index world and its tensor are specified:

```text
represent(U) star represent(V)
  ~=
represent(U tensor V)
```

State the reindexing and density-Day comparison map. External-product patches on `B1 x B2` do not by themselves establish this law.

### Spatial convolution continuity law

The two product projections preserve required halo labels, restrictions, and local/global identity under the effective representation.

### Spatial convolution effectivity law

External-product patch enumeration, halo construction, normalization, and invalidation remain within the declared resource bound.

## Relationship to existing mechanics

- **Behavioral coalgebra** models how behavior unfolds over time.
- **Comonad coalgebra** models how an object is coherently situated in local context.
- **Sheafification** glues compatible local meanings.
- **Density comonad** generates the locality structure from patches.
- **Day convolution** composes indexed/spatial descriptions over a base tensor.
- **Codensity probes** reconstruct large semantic behavior from observations.
- **Exact Context** compiles a task-indexed germ for semantic consumption.
- **Yoneda/Coyoneda** expose observations and generation paths.
- **Pullbacks** express agreement of local views; **pushouts** glue declared overlap.

These structures may combine, but do not conflate their laws.

## Guardrails

1. Not every comonad is a topological space. Its topological shadow may forget important multiplicity and local structure.
2. Not every dependency graph is a comonadic space. Name the endofunctor, counit, comultiplication, or use spatial language only as an approximation.
3. Not every cover is a basis. Require density/canonical reconstruction evidence.
4. A local point need not equal a global point. Preserve the map and provenance between them.
5. A point-preserving boundary need not be continuous.
6. Unlabelled halos may erase destinations and distinct witnesses of nearness; prefer labelled halos when architecture depends on edge meaning.
7. Infinite halos require an effective presentation, approximation policy, and resource model.
8. A product of point sets is not automatically the Day product of comonadic spaces.
9. Convolution of patch descriptions does not replace sheaf compatibility/gluing.
10. This is based on a 2026 v1 preprint. Treat software prescriptions as research-informed engineering hypotheses and preserve counterexamples.

## Selection rule

Use Comonadic Spatiality only when all are true:

```text
locality changes correctness or ownership;
points and local patches can be named;
a halo or basis changes code/test shape;
a locality-preservation law can be executed or approximated;
a finite/effective representation or obstruction can be supplied.
```

Use spatial Day convolution only when two such spatial worlds have a real patch/product composition, external-product patches change the artifact, and representable/continuity/resource laws can be witnessed.

Otherwise use ordinary dependency analysis, a graph, a context object, direct product, or plain boundary language.
