# Accessibility (A11Y) Checklist

Minimum accessibility standards for UX audits.

---

## The Big 4 (Check These First)

### 1. Keyboard Navigation

**Test:** Can you use the entire app with only keyboard?

```
Tab        → Move forward
Shift+Tab  → Move backward
Enter      → Activate
Escape     → Close/cancel
Arrow keys → Navigate within components
```

**Pass:**
- [ ] All interactive elements reachable via Tab
- [ ] Focus order is logical (left→right, top→bottom)
- [ ] Focus indicator visible (outline, highlight)
- [ ] No keyboard traps (can always Tab out)
- [ ] Skip links for repetitive navigation

**Fail examples:**
```
✗ Dropdown only opens on hover
✗ Modal can't be closed with Escape
✗ Custom button not focusable
✗ Focus invisible on dark backgrounds
```

---

### 2. Color Contrast

**Minimum ratios (WCAG AA):**
- Normal text: 4.5:1
- Large text (18pt+): 3:1
- UI components: 3:1

**Tools:**
- Chrome DevTools → Accessibility panel
- WebAIM Contrast Checker
- axe DevTools extension

**Pass:**
- [ ] Text readable on all backgrounds
- [ ] Links distinguishable from body text
- [ ] Form field borders visible
- [ ] Focus states have sufficient contrast

**Fail examples:**
```
✗ Light gray text on white (#999 on #fff = 2.85:1)
✗ Placeholder text too light
✗ Disabled state indistinguishable
```

---

### 3. Not Color-Only

**Principle:** Don't rely solely on color to convey information.

**Pass:**
- [ ] Error fields have icon AND red border
- [ ] Required fields have asterisk AND label
- [ ] Charts have patterns, not just colors
- [ ] Links underlined OR have other indicator

**Fail examples:**
```
✗ Only red border indicates error
✗ "Green means go, red means stop" (colorblind users?)
✗ Pie chart with 10 similar colors, no legend
```

---

### 4. Screen Reader Basics

**Test with:** VoiceOver (Mac), NVDA (Windows), or axe DevTools

**Pass:**
- [ ] Images have alt text (or alt="" if decorative)
- [ ] Form inputs have labels
- [ ] Buttons have accessible names
- [ ] Headings create logical outline
- [ ] ARIA used correctly (or not at all)

**Fail examples:**
```html
<!-- Bad: image without alt -->
<img src="logo.png">

<!-- Bad: icon button without label -->
<button><svg>...</svg></button>

<!-- Bad: input without label -->
<input type="email" placeholder="Email">

<!-- Bad: div pretending to be button -->
<div onclick="submit()">Submit</div>
```

**Fixed:**
```html
<img src="logo.png" alt="Company Logo">

<button aria-label="Close dialog"><svg>...</svg></button>

<label for="email">Email</label>
<input type="email" id="email">

<button onclick="submit()">Submit</button>
```

## Quick A11Y Audit Checklist

### Perceivable
- [ ] Text alternatives for non-text content
- [ ] Captions for video
- [ ] Content can be presented different ways
- [ ] Contrast ratios meet minimums

### Operable
- [ ] All functionality via keyboard
- [ ] Users have enough time
- [ ] No content that causes seizures
- [ ] Users can navigate and find content

### Understandable
- [ ] Text is readable
- [ ] Pages work predictably
- [ ] Users can avoid and correct mistakes

### Robust
- [ ] Compatible with assistive technologies
- [ ] Valid HTML
- [ ] Name, role, value for UI components

## CLI Accessibility

CLIs have different accessibility considerations:

### Screen Reader Compatibility
- [ ] Output is plain text (parseable)
- [ ] No reliance on color alone
- [ ] Progress indicators have text fallback
- [ ] Errors go to stderr, output to stdout

### Cognitive Accessibility
- [ ] Clear, consistent command structure
- [ ] Helpful error messages
- [ ] `--help` available everywhere
- [ ] Confirmation before destructive actions

### Example: Good CLI Output

```bash
# Good - parseable, no color-only info
$ br list
Status: 3 open, 2 closed

ID      STATUS  TITLE
TKT-1   OPEN    Fix login bug
TKT-2   CLOSED  Add dark mode
TKT-3   OPEN    Update docs

# Bad - relies on colors
$ br list
● Fix login bug      (● is green, ● is red... colorblind?)
● Add dark mode
```

## Automated Testing

### Tools
| Tool | Best For |
|------|----------|
| axe DevTools | Browser extension, quick scan |
| Lighthouse | CI/CD integration |
| pa11y | Command-line, CI/CD |
| WAVE | Visual evaluation |

### Limitations

Automated tools catch ~30% of issues. Manual testing required for:
- Keyboard navigation flow
- Screen reader experience
- Cognitive load
- Context-specific issues

## WCAG Levels

| Level | Meaning | Target |
|-------|---------|--------|
| **A** | Minimum | Must meet |
| **AA** | Standard | Should meet (legal requirement in many places) |
| **AAA** | Enhanced | Nice to have |

**For most projects, target WCAG 2.1 AA.**

---

## Common Fixes

| Issue | Fix |
|-------|-----|
| Missing alt text | Add descriptive alt, or alt="" for decorative |
| Low contrast | Use darker text or lighter background |
| No focus indicator | Add `:focus` styles |
| Click-only interaction | Add keyboard event handlers |
| Missing form labels | Use `<label>` with `for` attribute |
| Div as button | Use `<button>` element |
