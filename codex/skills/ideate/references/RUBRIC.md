# Idea Evaluation Rubric

Score each idea from 1-5 on the core criteria.

## Core criteria

| Criterion | What to ask |
|---|---|
| Robust | Does it handle important edge cases and failure paths? |
| Reliable | Will it work consistently, not just in the happy path? |
| Performant | Is speed / efficiency good enough for the user context? |
| Intuitive | Will users understand what it does and how it behaves? |
| User-friendly | Does it reduce frustration and improve the experience? |
| Ergonomic | Does it remove friction, steps, or cognitive load? |
| Useful | Does it solve a real, frequent, or painful problem? |
| Compelling | Will users actually care about it? |
| Accretive | Does it clearly add value without making the system worse? |
| Pragmatic | Is it realistic given the likely constraints? |

## Additional tie-breakers for plan seeds

Use these when several ideas score similarly:

| Tie-breaker | What to ask |
|---|---|
| Strategic fit | Does it reinforce the product's direction rather than distract from it? |
| Validation potential | Can we cheaply learn whether this is worth a full plan? |
| Leverage | Does it unlock later improvements or reduce future work? |

## Cut rules

Strong reasons to cut an idea:
- it solves a weak or unclear problem
- it duplicates existing work without a compelling delta
- it needs miracle assumptions
- it creates more complexity than value
- it cannot be validated until very late
- it is mostly an implementation detail disguised as strategy

## Recommended weighting

Weighted average:
- Useful: 2.0x
- Pragmatic: 2.0x
- Accretive: 1.5x
- Validation potential: 1.5x
- Strategic fit: 1.25x
- everything else: 1.0x

## Winnowing process

1. **Hard cuts**  
   Remove ideas with fatal flaws or obvious duplication.

2. **Threshold**  
   Remove ideas with poor weighted scores.

3. **Ranking**  
   Rank the remainder by weighted score and rationale quality.

4. **Synergy check**  
   Keep complementary ideas in view even if they are not #1 individually.

5. **Selection**  
   Pick the leading direction only after overlap and validation potential are considered.

## Red flags

Be skeptical of ideas justified mainly by:
- “it would be cool”
- “users can learn it”
- “we can document it later”
- “it’s technically possible”
- “it sounds more ambitious”
