# Kan lift architecture playbook

## Core reason lifts matter

Kan extensions help a system grow from a known core. Kan lifts help a system rebuild itself from its commitments.

Most mature codebase refactors are outside-in:

```text
public contracts, clients, traces, tests, reports, policies
        must remain true
internal modules, storage, services, workflows, runtimes
        must change
```

That is a lift-shaped problem:

```text
A --L?--> B
|         |
F         P
v         v
C
```

- `A`: required cases, user stories, public endpoints, workflows, tests, policies, report cases, or migration slices.
- `B`: internal implementation/design world: modules, services, workflows, state machines, repositories, handlers, resources, capabilities.
- `C`: observable behavior world: responses, emitted events, traces, persisted views, reports, policy outcomes, visible state.
- `P : B -> C`: fixed projection from internal implementation to observable behavior.
- `F : A -> C`: required observable behavior.
- `L : A -> B`: implementation-side candidate lift.

Architecture slogan:

```text
Kan extension = transport meaning across a boundary.
Kan lift      = synthesize or constrain what must exist behind a boundary.
```

Source basis for the postcomposition framing: `[KAN-NLAB-LIFT]`.

## The commitment inventory

Before choosing a lift, inventory external commitments. A commitment is anything a refactor must keep true.

Examples:

- public API responses and status codes;
- event payloads and ordering promises;
- audit traces and compliance reports;
- legacy client queries;
- UI-visible behavior;
- database views consumed externally;
- contract/golden tests;
- operational metrics or alert semantics;
- idempotency, authorization, security, and billing behavior.

Each commitment should become either:

- an object of `A`, if it is a requirement/test/use case;
- an object or observation in `C`, if it is externally visible behavior;
- part of `P`, if it describes how internals are observed.

Do not start with modules. Start with commitments.

## The projection `P` must be real

A lift is only useful if `P : B -> C` is a concrete architectural boundary.

Good `P` examples:

- running a service method and observing response/events;
- serializing an internal model to API DTOs;
- compiling a workflow to runtime traces;
- projecting database state to a public view;
- interpreting effectful programs into observable logs;
- rendering UI state to visible screens;
- evaluating internal policy rules to public allow/deny outcomes.

Bad `P` examples:

- “the codebase is observable somehow”;
- “the service should work”;
- “the repository is used by the controller”;
- any projection not implemented, testable, or centralizable.

Guardrail:

```text
No P, no lift.
```

## Left lift: implementation realization

Use a left-lift framing when the comparison cell is:

```text
η : F -> P · L
```

Engineering reading:

```text
The required behavior is covered, realized, or generated by projecting the internal implementation.
```

Typical prompts:

- “Given this public contract, synthesize the internal service/workflow design.”
- “Given these golden behaviors, refactor internals while preserving them.”
- “Given this desired trace, generate an effectful program/handler that produces it.”
- “Given this view update, find an internal source update that realizes it.”

Minimum realization test:

```text
for a in witness cases:
  required = F(a)
  actual   = P(L(a))
  assert required <= actual
```

The relation `<=` must be stated. It may mean exact equality, subset inclusion, contract containment, trace refinement, policy implication, or snapshot equivalence.

## Right lift: residual obligation discovery

Use a right-lift framing when the comparison cell is:

```text
ε : P · R -> F
```

Engineering reading:

```text
The projected internal obligations are sound with respect to the required behavior.
```

Typical prompts:

- “Given these contract tests, what internal obligations must exist?”
- “Given this compliance report, what fields/transitions/resources are required?”
- “Given this public behavior, what information must the domain model preserve?”
- “Given this API view, what source-side updates are sound?”

Minimum residual test:

```text
for a in witness cases:
  obligation = R(a)
  actual     = P(obligation)
  allowed    = F(a)
  assert actual <= allowed
```

Again, define the order before using words like weakest, strongest, least, greatest, sufficient, or sound.

## Exact, approximate, and impossible lifts

Every lift analysis must classify the result.

### Exact lift

```text
P(L(a)) == F(a)
```

Use when a candidate implementation projects exactly to the required behavior on the chosen observations.

### Covering / over-approximating lift

```text
F(a) <= P(L(a))
```

Use when the implementation provides at least the required behavior, possibly more. Common for generated capabilities or broad service implementations.

### Sound / under-approximating lift

```text
P(R(a)) <= F(a)
```

Use when derived obligations are guaranteed sound but may not be complete.

### No exact lift

No candidate in `B` projects to the required behavior in `C`.

This is often the most valuable output. Emit a no-exact-lift report instead of forcing a fake implementation.

Report:

```text
Required observation:
Current projection P cannot produce it because:
Missing internal data/capability/transition:
Possible repairs:
  1. enrich B
  2. change P
  3. weaken F
  4. add external dependency
  5. accept approximate lift with documented residual risk
Witness test that currently fails:
```

## Obligation ledger

For lift-shaped refactors, maintain an obligation ledger. Each row connects external behavior to internal requirements.

| External commitment | Observation in C | Needed internal artifact in B | Projection P path | Status | Witness test |
|---|---|---|---|---|---|
| endpoint returns cancellation reason | `ResponseBody.reason` | persisted `Cancellation.reason` | DTO serializer | missing | `cancel-order.reason.test` |
| refund is idempotent | `SecondCallSameResult` | idempotency key store | request handler trace | present | `refund.idempotency.test` |
| audit log emitted | `AuditEvent(Refunded)` | audit sink call | event trace projection | present | `refund.audit.test` |

Statuses:

- `present`: internal artifact exists and `P` exposes it;
- `missing`: no exact lift until artifact is added;
- `ambiguous`: several possible internal realizers;
- `external`: requires dependency outside current `B`;
- `weakened`: public behavior must be relaxed;
- `approximate`: only covering/sound approximation available.

## Defunctionalized lift IR

For implementation synthesis, make lift artifacts first-order:

```text
PublicObservation       -- Yoneda-like C-side observations
CandidateRealizer       -- Coyoneda-like B-side implementation payloads
ProjectionPath          -- defunctionalized P paths
ImplementationObligation -- residual R-side requirements
```

Typical functions:

```text
runObservation      : ObservableBehavior × PublicObservation -> Result
projectImplementation : CandidateRealizer × ProjectionPath -> ObservableBehavior
satisfyObligation   : InternalDesign × ImplementationObligation -> CheckResult
explainNoLift       : Requirement × Projection -> NoLiftReport
```

Law:

```text
runObservation(projectImplementation(realizer(a), path), obs)
  == runObservation(F(a), obs)
```

or, for approximate lifts, replace equality by the chosen comparison relation.

## Lift refactor loop

Use this loop for architecture-changing work:

1. Inventory external commitments.
2. Choose one witness case `a ∈ A`.
3. Define observable behavior `F(a)` precisely.
4. Identify or implement projection `P : B -> C`.
5. Attempt a candidate `L(a)` or residual `R(a)`.
6. Write the projection law test before broad module movement.
7. If no exact lift exists, emit obligations/repairs instead of overfitting.
8. Defunctionalize observations, projection paths, and realizers when they cross module boundaries.
9. Roll out only after the witness passes and the obligation ledger is stable.

## Failure modes

- The projection `P` is implicit, scattered, or bypassed by controllers/tests.
- The public behavior `F` is vague; no comparison relation is stated.
- The lift passes one endpoint but fails naturality across related cases in `A`.
- The system lacks information required to realize `F`; no-exact-lift should be reported.
- The implementation overbuilds without proving minimality or coverage.
- The residual obligation is too weak because observations in `C` were incomplete.
- The agent refactors modules before installing projection tests.
