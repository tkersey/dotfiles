# Kan lifts

## Why lifts belong beside extensions

Kan extensions and Kan lifts are adjoints to different composition operations.

Kan extensions solve a **precomposition** problem:

```text
K* : [D, E] -> [C, E]
K*(G) = G · K
Lan_K ⊣ K* ⊣ Ran_K
```

Kan lifts solve a **postcomposition** problem:

```text
P_* : [A, B] -> [A, C]
P_*(G) = P · G
Lft_P ⊣ P_* ⊣ Rft_P
```

when the relevant adjoints exist.

Architectural reading:

- Extension: “I know source semantics. Extend or restrict them across a boundary.”
- Lift: “I know desired observable behavior and a fixed boundary. Solve for the internal artifact behind that boundary.”

Sources: `[KAN-NLAB-LIFT]`, `[KAN-RIEHL-CTIC]`.

## Definitions used by this skill

Notation varies in the literature. This skill uses:

```text
Lft_P F   left Kan lift of F through P
Rft_P F   right Kan lift of F through P
P_*       postcomposition with P
```

Given:

```text
A --?--> B
|        |
F        P
v        v
C
```

where `P : B -> C` and `F : A -> C`.

### Left Kan lift

A left Kan lift of `F` through `P` is a functor:

```text
Lft_P F : A -> B
```

with a natural transformation:

```text
η : F -> P · Lft_P F
```

such that for every `G : A -> B`, composition with `η` gives a natural bijection:

```text
Nat(Lft_P F, G) ≅ Nat(F, P · G)
```

Engineering reading: a left lift is a universal realization candidate. It solves for an internal artifact whose projection can cover or realize the desired external behavior.

### Right Kan lift

A right Kan lift of `F` through `P` is a functor:

```text
Rft_P F : A -> B
```

with a natural transformation:

```text
ε : P · Rft_P F -> F
```

such that for every `G : A -> B`, composition with `ε` gives a natural bijection:

```text
Nat(G, Rft_P F) ≅ Nat(P · G, F)
```

Engineering reading: a right lift is a universal residual or obligation candidate. It solves for an internal artifact whose projection is sound with respect to the desired external behavior.

## Comparison-cell rule

Do not choose left vs right by intuition alone. Choose by the comparison cell the architecture needs.

```text
F -> P · implementation      left lift
P · implementation -> F      right lift
```

Examples:

- “Find an implementation that covers this public contract” usually wants `F -> P·L`.
- “Find internal obligations that are sufficient under this public spec” usually wants `P·R -> F`.
- “Run the already-known implementation through the public boundary” wants `P_*`, not a lift.

## Poset implementation recipe

Many architecture problems can be approximated as finite posets.

Let:

```text
A = discrete set of requirements/features/tests
B = poset of implementation choices
C = poset of observable guarantees
P : B -> C monotone
F : A -> C desired guarantee
```

If order means “less capable / less constrained / less informative than,” then pointwise approximations are:

```text
Lft_P F(a) = least b in B such that F(a) <= P(b)
Rft_P F(a) = greatest b in B such that P(b) <= F(a)
```

Use this only after stating the order. Reverse the inequalities if the order means the opposite, for example “more restrictive than.”

Tests:

```text
F(a) <= P(Lft_P F(a))
P(Rft_P F(a)) <= F(a)
```

Minimality/maximality tests:

```text
no smaller b covers F(a)       for Lft
no larger b remains sound      for Rft
```

## Relational/residual intuition

In relation-like settings, right lifts behave like residuals or weakest-precondition operators. A common Boolean-enriched shape is:

```text
R(a, b) holds iff for every observable c,
  P(b, c) implies F(a, c)
```

This is useful for policy obligations, query containment, test-suite obligations, and view-update constraints. Check the relation orientation before using this formula in code.

Source: `[KAN-NLAB-LIFT]`.

## Architectural patterns

### Public contract to internal service design

- `A`: endpoint/feature category.
- `B`: internal service/module design options.
- `C`: observable API behavior.
- `P : B -> C`: public API projection of an internal design.
- `F : A -> C`: desired public contract.
- Candidate: `Lft_P F`.
- Test: every desired endpoint behavior is covered by projecting the synthesized internal design.

### Tests to residual implementation obligations

- `A`: test/spec category.
- `B`: implementation obligations or capability sets.
- `C`: observable behavior lattice.
- `P : B -> C`: behavior generated by obligations.
- `F : A -> C`: required behavior.
- Candidate: `Rft_P F` when soundness is `P·impl -> F`.
- Test: every generated behavior is accepted by the spec, and obligations are maximal/weakest under the chosen order.

### View update / reverse migration

- `B`: source/internal database instances.
- `C`: public views or target schema observations.
- `P : B -> C`: view/projection/query functor.
- `F : A -> C`: desired public view change.
- Candidate: lift through `P`.
- Test: projecting the lifted internal update realizes or soundly approximates the desired view update.

### Effect handler inverse design

- `B`: effectful programs or handler strategies.
- `C`: observable runtime traces.
- `P : B -> C`: interpreter/handler trace semantics.
- `F : A -> C`: desired trace contract.
- Candidate: lift through `P`.
- Test: interpreted lifted program produces traces related to the desired trace contract by the chosen comparison cell.

## Failure modes

- The boundary `P` is not actually fixed; then the problem is ordinary design, not a Kan lift.
- The unknown is after the boundary; then use `Lan`/`Ran`, not a lift.
- The order/2-cell direction is unstated; then “least,” “greatest,” “weakest,” and “strongest” are meaningless.
- The pointwise choices do not assemble functorially across morphisms in `A`.
- The synthesized implementation passes the witness but bypasses the central projection `P`.
